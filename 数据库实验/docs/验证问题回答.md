# 实验验证问题逐条作答

以下 1~8 对应实验指导书"三、实验验证"中的 8 个问题。答案针对本次使用的 MySQL 8.x 给出，并在需要时标明与 SQL 标准 / 其他 DBMS 的差异。

## 1. 分组查询中，不在 GROUP BY 子句中出现的属性，是否可以出现在 SELECT 子句中？

**结论：不能直接出现，除非满足以下三种情况之一：**

1. 该属性被包含在聚合函数里，如 `AVG(Grade)`、`MAX(Sage)`；
2. 该属性与 `GROUP BY` 中的某个列存在函数依赖（例如 `GROUP BY Sno`，而 Sno 是主键，则同表其他列都可以随意出现）——SQL 标准与 MySQL 都允许；
3. 显式使用 `ANY_VALUE(col)` 告诉 DBMS"任取一值即可"。

**MySQL 附注**：默认 `sql_mode` 中含 `ONLY_FULL_GROUP_BY`，违反时直接报 `ERROR 1055`。若关闭此模式，MySQL 会允许但返回的非聚合列值是"不确定"的，不推荐使用。

**验证片段**（见 `sql/04_queries.sql` B1）：

```sql
-- 合法：Sdept 在 GROUP BY 中
SELECT Sdept, COUNT(*), AVG(Sage) FROM Student GROUP BY Sdept;

-- 非法：Sname 不在 GROUP BY 且不在聚合中
-- SELECT Sdept, Sname, COUNT(*) FROM Student GROUP BY Sdept;  -- ERROR 1055
```

## 2. 建立视图时，组成视图的属性可以部分指定吗？总是可以全部省略吗？

**结论：**

- **不能部分指定**：视图列名要么全部省略，要么全部显式列出，二者不可混合（SQL 标准规定，MySQL、Oracle、SQL Server 均如此）。
- **不总是可以全部省略**：以下情况必须显式给出视图列名
  1. `SELECT` 中有表达式或函数，生成的列没有原生名字；例如 `SELECT Sno, AVG(Grade) FROM SC GROUP BY Sno`；
  2. 多表连接时列名发生冲突（如 `Student.Sno` 与 `SC.Sno` 都叫 `Sno`）；
  3. 需要对列进行重命名以隐藏基表的内部列名。

**验证片段**（见 `sql/06_views.sql`）：

```sql
-- 全部省略（可行）
CREATE VIEW V_IS_Student AS
SELECT Sno, Sname FROM Student WHERE Sdept='IS';

-- 必须全部指定（含聚合）
CREATE VIEW V_Student_AvgGrade (Sno, Sname, AvgGrade, CourseCnt) AS
SELECT s.Sno, s.Sname, AVG(sc.Grade), COUNT(sc.Cno)
FROM Student s LEFT JOIN SC sc ON s.Sno=sc.Sno
GROUP BY s.Sno, s.Sname;
```

## 3. 可以通过视图将属性值修改为不满足视图定义的值吗？

**结论：默认情况下可以；带 `WITH CHECK OPTION` 则不可以。**

- 默认情形：视图不是存储结构，只是查询；通过它 `UPDATE/INSERT` 基表时，DBMS 不会检查新值是否仍在视图视野内，因此你可以把"IS 系学生"改成"MA 系学生"，其结果是该行随之"消失"于视图。
- 加上 `WITH CHECK OPTION` 后，DBMS 会在每次 UPDATE/INSERT 前验证结果是否仍满足视图的 `WHERE` 条件，违反则报错 `ERROR 1369 (HY000): CHECK OPTION failed`。

**验证片段**（见 `sql/06_views.sql` C3 ~ C4）：

```sql
-- 被拒绝
UPDATE V_IS_Student_Check SET Sdept='MA' WHERE Sno='201515004';
-- 被拒绝
INSERT INTO V_IS_Student_Check (Sno,Sname,Sdept) VALUES ('201515096','Yi Qian','CS');
```

## 4. 视图都是可更新的吗？

**结论：不是。** 一般只有基于单表、不含聚合 / 去重 / 复杂子查询的视图才可更新。

MySQL 对不可更新视图的常见情形（命中任一条即不可更新）：

- 含 `DISTINCT`；
- 含聚合函数（`AVG / SUM / COUNT / MAX / MIN`）；
- 含 `GROUP BY / HAVING`；
- 含 `UNION / UNION ALL`；
- 在子查询中引用被更新表；
- 某些多表连接视图只能更新"可直接修改的基表那一侧"；
- 基于不可更新视图构造的视图；
- 视图列是字面量、表达式、函数结果。

**验证片段**（见 `sql/06_views.sql` C2）：尝试 `UPDATE V_Student_AvgGrade` 会返回 `ERROR 1288: The target table V_Student_AvgGrade of the UPDATE is not updatable`。

## 5. 删除基本表时会自动删除由该基本表导出的视图吗？

**结论：不会自动删除，视图会变为"失效/无效"。**

- MySQL 行为：`DROP TABLE` 不报错、也不级联删除视图，但视图进入失效状态；再 `SELECT` 该视图时会报 `ERROR 1356: View '...' references invalid table(s)...`。需要管理员手动 `DROP VIEW`。
- Oracle：同样不会自动删除；视图被标记为 INVALID，可在基表重建后通过 `ALTER VIEW ... COMPILE` 重新编译。
- SQL Server：`DROP TABLE` 同样不会删除依赖视图，视图在执行时才报错。

`DROP TABLE` 语法上不支持 `CASCADE` 到视图（不同于 PostgreSQL 的 `DROP TABLE ... CASCADE` 会级联删除依赖视图）。

## 6. WITH GRANT OPTION 和 WITH ADMIN OPTION 有什么区别？

两者的共同点：都表示"允许被授权者将该权限再授予他人"。差别主要体现在作用的权限类别：

| 维度 | WITH GRANT OPTION | WITH ADMIN OPTION |
| ---- | ----------------- | ----------------- |
| 适用对象 | 对象权限（SELECT / UPDATE / INSERT / DELETE / EXECUTE 等） | 系统权限和角色（CREATE TABLE、CREATE USER、ROLE 等） |
| 再授权的语义 | 把同样的对象权限授予别人 | 把同样的系统权限或角色授予别人 |
| 回收行为 | 回收 GRANT OPTION 一般不回收已授予的下游权限（SQL 标准允许 `CASCADE` 回收） | 回收 ADMIN OPTION 时，下游已授出的同一角色 / 系统权限一般也连带回收 |
| 标准 | SQL 标准中定义 | Oracle / SQL:1999 定义，用于角色 |

**各 DBMS 的落地情况：**

- **Oracle**：两者**严格分离** —— 对象权限用 `WITH GRANT OPTION`，系统权限和角色用 `WITH ADMIN OPTION`。
- **PostgreSQL**：对象权限 `WITH GRANT OPTION`；角色则通过 `GRANT role TO user WITH ADMIN OPTION`。
- **MySQL**：**只有 `WITH GRANT OPTION` 一种语法**，无论对象权限还是角色再授权一律写 `WITH GRANT OPTION`。MySQL 的角色授予 `GRANT r_read TO u1 WITH ADMIN OPTION` 语法不被支持；若要让 u1 能把角色授予别人，需执行 `GRANT r_read TO u1 WITH ADMIN OPTION` 的**概念**在 MySQL 中由 `WITH GRANT OPTION` 统一承担。

**验证片段**（见 `sql/07_privileges.sql`）：

```sql
GRANT ALL PRIVILEGES ON scs.* TO 'u_admin'@'localhost' WITH GRANT OPTION;
-- 登录 u_admin 后可再执行：
-- GRANT SELECT ON scs.Teacher TO 'u_student'@'localhost';
```

## 7. 验证数据库管理系统维护数据库完整性的功能

DBMS 的三类完整性及其保护机制：

1. **实体完整性**（主键非空且唯一）
   - 机制：`PRIMARY KEY` / `UNIQUE` 约束
   - 验证（见 `08_constraints.sql` B1）：插入重复 Sno 会被拒绝：`Duplicate entry '201515001' for key 'Student.PRIMARY'`。
2. **参照完整性**（外键）
   - 机制：`FOREIGN KEY ... ON DELETE / ON UPDATE`
   - 验证（B5 / B6）：向 SC 插入不存在的 Sno 会被拒绝；删除 Course 中仍被 SC 引用的记录会被拒绝。
3. **用户定义完整性**（`NOT NULL` / `CHECK` / `DEFAULT` / `UNIQUE`）
   - 机制：MySQL 8.0.16 起支持 `CHECK`；8.0.16 之前 CHECK 只是语法糖被忽略。
   - 验证（B2 / B3）：不提供 `Sdept` 被 `NOT NULL` 拒绝；`Ssex='X'` 因列为 `ENUM('M','F')` 而被拒绝（非法枚举值）。

此外，触发器也可以看作"用户定义完整性"的扩展，用于表达更复杂的业务规则（见 `09_triggers.sql` 的 BEFORE INSERT/UPDATE 截断 Grade 的用例）。

## 8. 其他想要验证的问题

本实验中额外进行的验证：

### 8.1 触发器实现不能用 CHECK 表达的业务规则

CHECK 只能引用同一行的列。要实现"学生某学期选课数不超过 6 门"这类跨行约束，只能通过触发器：

```sql
CREATE TRIGGER trg_sc_limit
BEFORE INSERT ON SC
FOR EACH ROW
BEGIN
    DECLARE cnt INT;
    SELECT COUNT(*) INTO cnt FROM SC WHERE Sno = NEW.Sno;
    IF cnt >= 6 THEN
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT='每位学生最多选 6 门课';
    END IF;
END;
```

### 8.2 级联删除与 SET NULL 的区别

Student → SC 外键指定 `ON DELETE CASCADE`，删除学生时其选课记录自动删除；而 Student → Student 自引用外键指定 `ON DELETE SET NULL`，删除导师时被导学生的 `Tutor_Sno` 被置空。这展示了 DBMS 在维护参照完整性时的灵活性。

### 8.3 事务与回滚

整个 `05_updates.sql` / `06_views.sql` / `08_constraints.sql` / `09_triggers.sql` 都使用 `START TRANSACTION; ... ROLLBACK;` 来演示"任意更新都可以安全回滚"。这验证了 DBMS 通过事务日志保证了原子性与一致性。

### 8.4 `sql_mode` 对查询的影响

同一条 SQL 在 `ONLY_FULL_GROUP_BY` 开启 / 关闭时表现不同。这提醒我们：应用系统应显式设置 `sql_mode` 以避免环境差异导致的"玄学"bug。
