## 三、数据库设计

### 3.1 场景描述

模拟一所大学的"学生-教师-课程-选课"信息管理系统（简称 SCS）。系统维护：

- **学生**（Student）：可有高年级学生担任导师（Tutor_Sno 自引用）；
- **教师**（Teacher）：每位教师可讲授多门课程；
- **课程**（Course）：课程之间存在先修关系（Cpno 自引用），每门课由一位教师讲授；
- **选课**（SC）：记录学生选课及成绩。

### 3.3 关系模式

- `Student (Sno, Sname, Ssex, Sage, Sdept, Tutor_Sno)`，主键 `Sno`，`Tutor_Sno` 自引用外键；
- `Teacher (Tno, Tname, Ttitle, Tdept)`，主键 `Tno`；
- `Course (Cno, Cname, Cpno, Ccredit, Tno)`，主键 `Cno`，`Cpno` 自引用，`Tno` 外键；
- `SC (Sno, Cno, Grade)`，联合主键 `(Sno, Cno)`，`Sno` / `Cno` 外键。

完整性约束：主键、外键、`NOT NULL`、`UNIQUE`、`CHECK`（详见 `sql/01_schema.sql`）。

### 4.6 权限管理

见 `sql/07_privileges.sql`。

**u_student（只读角色 r_read_only）：** 登录后可以 `SELECT * FROM Student` 读取数据，但尝试 `UPDATE SC` 时返回 `ERROR 1142 (42000): UPDATE command denied`，说明只读权限生效。

![u_student 可查询，UPDATE 被拒绝](images/22.png)

**u_teacher（成绩录入角色 r_grade_writer）：** 登录后可以读 `Student / Course`，并被允许 `UPDATE SC SET Grade=...`（列级授权只开放 `Grade` 列）；但尝试 `UPDATE Student SET Sage` 时同样返回 `ERROR 1142`，体现了"最小权限"原则。

![u_teacher 可改 SC.Grade，不能改 Student](images/23.png)

**u_admin（scs 库的 DBA，`WITH GRANT OPTION`）：** 登录后成功执行 `GRANT SELECT ON scs.Teacher TO 'u_student'@'localhost'`，说明其具备授权传递能力；这正是 `WITH GRANT OPTION` 的效果。

### 4.7 三类完整性约束

见 `sql/01_schema.sql`（建表时）和 `sql/08_constraints.sql`（建表后）。

下图汇总了 `08_constraints.sql` 中一组"故意违约"用例的报错信息，每条报错都印证了 DBMS 会主动维护相应的完整性：

- `ERROR 1062 Duplicate entry '201515001' for key 'student.PRIMARY'` —— **实体完整性**：主键重复被拒绝；
- `ERROR 1364 Field 'Sdept' doesn't have a default value` —— **域完整性/非空约束**：`NOT NULL` 列未给值；
- `ERROR 1265 Data truncated for column 'Ssex'` —— **域完整性**：`ENUM('M','F')` 限制了取值；
- `ERROR 1062 Duplicate entry 'Wang Wei' for key 'teacher.uk_teacher_name'` —— **唯一约束** `UNIQUE` 生效；
- `ERROR 1452 Cannot add or update a child row: foreign key constraint fails ... fk_sc_student` —— **参照完整性**：插入孤立子表记录被拒绝；
- `ERROR 1451 Cannot delete or update a parent row: foreign key constraint fails ... fk_sc_course` —— **参照完整性**：删除被引用的父表记录被拒绝。

![六类违约均被 DBMS 拒绝](images/25.png)

## 五、实验结论

通过本次实验，系统性完成了：

1. 数据定义（DDL）：建库、建表、修改和删除表结构，同时体会了三类完整性约束是如何在建表时被声明的；
2. 数据操纵（DML）：掌握了各种形态的 SELECT（单表、连接、嵌套、集合）以及 INSERT/UPDATE/DELETE；
3. 视图：理解视图的"虚表"本质，验证了视图可更新性的约束条件；
4. 权限与安全：使用 MySQL 8 的角色机制实现分级授权，掌握 GRANT / REVOKE / WITH GRANT OPTION 的用法；
5. 完整性：通过违反约束的测试用例观察到 DBMS 对数据完整性的自动维护；
6. 触发器：利用 BEFORE/AFTER 触发器实现业务规则自动化；
7. 备份与恢复：能够使用 `mysqldump` 完成逻辑备份与恢复。

## 六、存在的问题与解决办法

> 以下是常见问题清单，同学可根据自己的实际情况保留 / 修改 / 补充。

1. **中文乱码**：建库时显式指定 `utf8mb4`，并在客户端设置 `SET NAMES utf8mb4;`。
2. **GROUP BY 报错 `ONLY_FULL_GROUP_BY`**：MySQL 8 默认 `sql_mode` 含该值。保证非聚合列都在 `GROUP BY` 中，或使用 `ANY_VALUE()`。
3. **DELETE 的子查询不能引用被删表**：MySQL 要求在子查询外包一层派生表绕过，见 `05_updates.sql` 的 B2。
4. **角色未默认激活**：登录后需 `SET ROLE ALL;` 或事先 `SET DEFAULT ROLE ALL TO <user>;`。
5. **DROP TABLE 后视图失效**：MySQL 不会级联删除视图，只能手动 `DROP VIEW`。

## 七、感想

本次实验把课堂上的关系模型和 SQL 语法落到了具体操作上，从建表、查询、更新到视图、权限、约束和触发器，一步步走完一条完整的数据管理链路，比单纯看书印象深得多。动手过程中最直观的体会是：只要把主键、外键、非空、唯一等约束设计清楚，数据库就能在后台替你挡住大量错误数据；再配合触发器，一些业务规则（如成绩范围、删除留痕）可以自动执行，不必全靠应用层反复校验。

权限与角色部分让我认识到，数据库不仅是存数据的容器，也是安全边界——不同用户该看什么、能改什么，应当用 GRANT 明确下来，而不是所有人都用 root。实验中也遇到过环境类问题（例如 PowerShell 重定向、`mysql` 未加入 PATH、客户端字符集与脚本编码不一致等），排查过程虽然耗时，但反过来加深了对“工具链”和“连接参数”的理解。

整体而言，这次实验让我对 MySQL 的基本操作有了系统信心，也意识到规范设计与约束先行的重要性；后续若有机会，还想在更复杂的事务与并发场景里继续练习。
